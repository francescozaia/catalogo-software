5. Metodologia

Pipeline dati

  1. Catalogo: crawler.py interroga l'API ufficiale api.developers.italia.it/v1/software con paginazione cursor-based (25 record/pagina, ~1 req/s). Per ogni record viene parsato il publiccode.yml.
  2. Metriche: metrics.py processa i repository GitHub con una singola query GraphQL per repo (tramite token gh), catturando: cronologia commit del default branch, ultime 100 PR e issue, release, struttura root e .github/workflows.
  3. Dashboard: build_dashboard.py arricchisce le metriche con i metadati del catalogo (IPA, ente, descrizione, status) e genera questo HTML standalone con dati embedded.

Finestra temporale

Salvo dove diversamente indicato (es. last_commit_date, oldest_open_issue_days che sono assoluti), tutte le metriche di attività si riferiscono alla finestra di 90 giorni antecedenti l'esecuzione del crawl. La scelta riflette un orizzonte rilevante per il software di produzione: né troppo breve da catturare il rumore, né troppo lungo da nascondere abbandoni recenti.

Score composito (0–100, modificabile dal pannello)

Quattro sub-score 0–100, combinati come media pesata. I pesi di default sono 25/25/25/25; l'utente può modificarli con gli slider per far emergere repo diversi a seconda dell'angolo di analisi.

  • Freshness: giorni dall'ultimo commit (soglia a 14/30/90/180/365 gg). archived = 0.
  • Velocity: 60% volume di commit in finestra + 40% lead time mediano delle PR mergeate. Premia chi spedisce frequentemente e rapidamente.
  • Quality: 40% review coverage + 30% presenza di CI + 30% bus factor (penalizza la concentrazione su singolo autore).
  • Maintenance: 25% età della issue aperta più vecchia + 25% tasso chiusura issue + 25% freschezza delle dipendenze (Dependabot/file di lock presenti) + 25% recency dell'ultima release.

Limiti noti

  • Multi-provider: GitHub, GitLab (gitlab.com + self-hosted) e Bitbucket sono supportati. Su Bitbucket alcune metriche (release, tasso chiusura issue, license) non sono esposte dall'API e vengono lasciate vuote.
  • Review coverage GitLab: usa user_notes_count > 0 come proxy (commenti sulla MR). Senza autenticazione l'API delle approvals non è disponibile.
  • Troncamento a 100: la query GraphQL/REST prende le 100 PR/issue più recenti. Per progetti molto attivi (improbabili in PA) il conteggio in finestra è un lower bound.
  • Bus factor: calcolato sui soli autori della finestra di 90 giorni, non sull'intera storia del repo.
  • Repo non più raggiungibili: il catalogo include URL che restituiscono 404 (repo cancellati o rinominati) o 403 (repo resi privati): segnalati come errore nel pannello.

Riproducibilità

Per rigenerare tutto: python3 crawler.py && python3 metrics.py && python3 build_dashboard.py. Lo script metrics.py richiede gh auth login già fatto.
